JIFAS Knowledge Base

Dokumen ini adalah knowledge base untuk AI Chatbot JIFAS. Isinya dibuat agar AI memahami logika bisnis, alur kerja, istilah, error handling, dan proses support untuk modul Account dan Accounting. Fokus dokumen ini bukan menjelaskan source code, tetapi menjelaskan cara sistem bekerja dari sisi user, finance, accounting, auditor, dan IT support.

Dokumen ini dapat dipakai untuk RAG, internal chatbot, user guide, training, dan troubleshooting.

Versi: 1.0
Cakupan utama: Account, Authentication, Authorization, Journal Transaction, Journal Memorial, CAJE PAJE, Adjust Journal, Reverse Journal, Bulk Posting, Balance Sheet, Cash Flow, Profit and Loss, Ledger, Trial Balance, Aging AP, Aging AR, dan Sales Receive.


1. Prinsip Jawaban AI Chatbot JIFAS

AI Chatbot JIFAS harus menjawab seperti support internal yang memahami finance dan accounting. Jawaban harus mudah dipahami user biasa, bukan terlalu teknis. Kalau masalah berhubungan dengan hak akses, akun, role, token, server, API, company mapping, menu tidak muncul, atau data tidak bisa dimuat, arahkan user untuk menghubungi IT JIFAS.

AI harus membedakan masalah yang bisa diselesaikan user dan masalah yang harus ditangani IT.

Masalah yang bisa dibantu user:
- Salah memilih periode laporan.
- Belum memilih company.
- Belum klik Preview.
- Total debit dan credit belum sama.
- Deskripsi belum diisi.
- Vendor atau tenant belum dipilih padahal akun membutuhkan party.
- Belum memilih COA.
- Belum memilih dokumen sebelum posting.
- Report kosong karena parameter belum lengkap.

Masalah yang harus diarahkan ke IT:
- User tidak bisa login padahal username dan password benar.
- Akun terkunci, tidak aktif, atau password Windows bermasalah.
- Menu tidak muncul atau access denied.
- Role, company code, atau menu access salah.
- Token expired terus menerus.
- API gagal dimuat, server error, atau data company tidak muncul.
- Master data vendor, tenant, COA, division, department, atau company tidak tersedia.
- Active period perusahaan tidak tersedia.
- Error request failed, unauthorized, forbidden, internal server error.
- Sistem gagal mengambil data jurnal saat reverse atau adjust.
- Posting gagal karena error sistem.
- Send to consolidation gagal karena error sistem.

AI harus selalu meminta informasi penting saat troubleshooting:
- Username user.
- Company yang dipilih.
- Menu yang sedang dibuka.
- Nomor dokumen atau nomor jurnal.
- Periode atau tanggal yang digunakan.
- Screenshot error.
- Waktu kejadian.
- Apakah user lain mengalami hal yang sama.

AI tidak boleh menyarankan user mengubah database, menghapus jurnal langsung, mem-bypass approval, atau mengubah data financial tanpa prosedur resmi.


2. Gambaran Umum JIFAS

JIFAS adalah Jababeka Integrated Finance and Accounting System. Sistem ini dipakai untuk mengelola proses keuangan internal Jababeka secara terpusat. JIFAS membantu perusahaan mencatat transaksi, melakukan approval, mengontrol budget, memproses pembayaran, mencatat kas dan bank, serta menghasilkan laporan keuangan.

JIFAS dapat dianalogikan sebagai gerbang kontrol keuangan perusahaan. Uang keluar dan uang masuk harus melewati proses pencatatan, verifikasi, approval, dan posting agar data keuangan akurat dan bisa diaudit.

Alur besar uang keluar:
Kebutuhan divisi masuk ke SPK atau kontrak, lalu invoice vendor diproses, disetujui head, dicek tax, disetujui finance, dibayar, dicatat di cash bank, lalu diposting ke accounting.

Alur besar uang masuk:
Customer membayar, sales receive dicatat, receiving cash bank dibuat, lalu transaksi diposting ke accounting.

Alur besar uang muka karyawan:
User mengajukan PUM, head menyetujui, finance dan tax memproses, pembayaran dilakukan, user melakukan settlement, lalu accounting memposting transaksi.

Semua transaksi pada akhirnya bermuara ke accounting report seperti Trial Balance, Balance Sheet, Profit and Loss, Cash Flow, Ledger, dan Aging Report.


3. Modul Account

#3.1 Fungsi Modul Account

Modul Account adalah pintu masuk utama ke JIFAS. Semua user harus login sebelum bisa membuka menu JIFAS. Modul ini memastikan hanya user yang memiliki akun, role, company mapping, dan menu access yang benar yang dapat masuk dan menggunakan sistem.

Fungsi utama Account:
- Memverifikasi identitas user.
- Mengelola sesi login.
- Menghubungkan user dengan role dan company.
- Mengontrol akses menu.
- Mengamankan data finance dan accounting.
- Menjadi dasar audit trail karena setiap transaksi harus punya pembuat atau pemroses yang jelas.


#3.2 Login

Login adalah proses user masuk ke JIFAS dengan username dan password. Pada lingkungan JIFAS, login umumnya menggunakan akun perusahaan atau akun Windows Active Directory. User tidak membuat akun sendiri seperti aplikasi publik.

Alur login:
1. User membuka halaman login JIFAS.
2. User mengisi username.
3. User mengisi password.
4. User dapat memilih Remember Me jika ingin sesi login bertahan lebih lama.
5. User menekan Sign in.
6. Sistem membersihkan local storage browser untuk membuang cache sesi lama.
7. Sistem mengirim username dan password ke authentication service.
8. Jika data benar, sistem membuat sesi login dan token akses.
9. Sistem memuat role, company, dan menu access user.
10. User diarahkan ke halaman utama JIFAS.

Jika login gagal, user tetap berada di halaman login dan sistem menampilkan pesan error.


#3.3 Local Storage Cleanup Saat Login

Saat halaman login dibuka dan saat tombol Sign in diklik, browser membersihkan local storage. Tujuannya bukan menghapus data transaksi JIFAS, tetapi membersihkan cache pada browser.

Manfaatnya:
- Menghindari konflik company lama.
- Menghindari menu lama masih terbaca setelah user berbeda login.
- Mengurangi risiko data user sebelumnya terbawa di komputer bersama.
- Memaksa sistem mengambil ulang role, company, dan menu terbaru.

Jika user mengalami menu aneh, company tidak sesuai, atau data tidak berubah setelah ganti user, arahkan user logout dan login ulang. Jika tetap terjadi, arahkan ke IT.


#3.4 Remember Me

Remember Me adalah opsi agar sesi login bertahan lebih lama di browser. Jika dipilih, user tidak perlu login ulang terlalu sering. Jika tidak dipilih, sesi akan lebih cepat habis dan user perlu login ulang setelah idle atau browser ditutup.

Jika komputer dipakai bersama, sebaiknya Remember Me tidak digunakan demi keamanan.


#3.5 Role dan Hak Akses

Setelah login berhasil, sistem membaca role user. Role menentukan menu yang bisa dibuka dan aksi yang bisa dilakukan.

Contoh role:
- USER:KI adalah user biasa di perusahaan KI.
- HEAD:KI adalah head atau atasan di perusahaan KI.
- FINA:KI adalah staff finance di perusahaan KI.
- HDFA:KI adalah head finance di perusahaan KI.
- ACCT:KI adalah staff accounting di perusahaan KI.
- HDAC:KI adalah head accounting di perusahaan KI.
- TAXS:KI adalah staff pajak di perusahaan KI.
- WMTR:RC adalah webmaster untuk perusahaan RC.
- ADMN:KI adalah admin sistem untuk perusahaan KI.

Format role umumnya berbentuk kode peran dan kode perusahaan. Contoh FINA:KI berarti finance untuk company KI.

Jika user bisa login tetapi menu tidak muncul, atau user tidak bisa membuka menu tertentu, kemungkinan role atau menu access belum sesuai. Ini harus dicek oleh IT atau administrator JIFAS.


#3.6 Company Code dan Multi Company

JIFAS dipakai oleh banyak perusahaan dalam grup Jababeka. User biasanya hanya dapat melihat data sesuai company code yang diberikan.

Contoh:
- User dengan company KI hanya melihat data KI.
- User dengan company RC hanya melihat data RC.
- Modul konsolidasi dapat menggabungkan beberapa company, tetapi aksesnya dibatasi.

Jika user merasa data tidak sesuai company atau data company tidak muncul, kemungkinan penyebabnya:
- User memilih company yang salah.
- User belum memiliki mapping company.
- Company code di session browser bermasalah.
- Hak akses company belum diberikan.
- API company gagal dimuat.

Jika sudah login ulang tetapi masih salah, arahkan ke IT.


#3.7 Forgot Password

Forgot Password dipakai ketika user lupa password. User mengisi email, lalu sistem mengirim link reset password jika email terdaftar.

Alur forgot password:
1. User memilih Forgot Password.
2. User mengisi email.
3. Sistem memvalidasi email.
4. Jika valid, sistem mengirim email reset.
5. User membuka email dan klik link reset.
6. User membuat password baru.
7. User login kembali.

Jika email reset tidak masuk, user harus cek spam atau junk mail. Jika tetap tidak ada, arahkan ke IT karena bisa jadi email tidak terdaftar, mail server bermasalah, atau akun user dikelola oleh Active Directory sehingga reset tidak melalui halaman JIFAS.


#3.8 Reset Password

Reset Password dipakai setelah user menerima link reset. User mengisi email, password baru, dan confirm password.

Validasi umum reset password:
- Email harus diisi.
- Password harus diisi.
- Confirm password harus sama dengan password.
- Link atau kode reset harus valid.
- Link reset tidak boleh expired.

Jika link expired, user harus melakukan forgot password ulang. Jika password tetap tidak bisa direset, arahkan ke IT.


#3.9 Email Confirmation

Email Confirmation adalah proses memastikan email user benar. Setelah email dikonfirmasi, sistem dapat mengizinkan user login atau menyelesaikan proses akun.

Pada sistem internal perusahaan, fitur ini bisa ada karena template standar ASP.NET, tetapi tidak selalu digunakan sebagai proses utama. Jika user bingung tentang email confirmation, arahkan ke IT untuk memastikan status akun.


#3.10 External Login

External Login memungkinkan login menggunakan provider lain seperti Microsoft, Google, atau OAuth provider perusahaan. Jika tidak ada provider yang dikonfigurasi, sistem akan menampilkan bahwa external authentication service belum tersedia.

Jika user memilih external login dan gagal, kemungkinan penyebabnya:
- Provider belum dikonfigurasi.
- Akun external tidak terhubung dengan akun JIFAS.
- Token provider expired.
- Koneksi provider bermasalah.
- Akses user belum dimapping.

Solusi: arahkan ke IT JIFAS.


#3.11 Two Factor Authentication

Two Factor Authentication atau 2FA adalah verifikasi tambahan setelah password. Sistem dapat meminta user memilih provider, lalu memasukkan kode verifikasi.

Alur 2FA:
1. User login dengan username dan password.
2. Sistem meminta kode verifikasi.
3. User memilih provider verifikasi.
4. Sistem mengirim kode.
5. User memasukkan kode.
6. Jika kode valid, user berhasil login.

Remember Browser dapat dipilih agar browser dipercaya dan tidak selalu meminta kode ulang.

Jika kode salah atau expired, user perlu meminta kode baru. Jika tetap gagal, arahkan ke IT.


#3.12 Register User

Register adalah fitur untuk membuat akun baru. Dalam lingkungan internal JIFAS, pembuatan akun biasanya tidak dilakukan bebas oleh user. Akun lebih aman dibuat atau dimapping oleh IT.

Jika user meminta dibuatkan akses JIFAS, AI harus mengarahkan user untuk mengajukan ke IT atau administrator JIFAS dengan informasi:
- Nama lengkap.
- Username Windows.
- Email perusahaan.
- Company.
- Role yang dibutuhkan.
- Menu yang dibutuhkan.
- Approval dari atasan jika diperlukan.


#3.13 Troubleshooting Account

Masalah: Tidak bisa login.
Kemungkinan penyebab:
- Username salah.
- Password salah.
- Password Windows expired.
- Akun terkunci.
- Akun belum dimapping ke JIFAS.
- Server authentication bermasalah.

Jawaban AI:
Pastikan username dan password sudah benar. Coba login ulang dan pastikan Caps Lock tidak aktif. Jika akun Windows juga bermasalah atau tetap gagal, hubungi IT JIFAS dengan screenshot error.

Masalah: Menu tidak muncul setelah login.
Kemungkinan penyebab:
- Role belum diberikan.
- Menu access belum dimapping.
- Company code tidak sesuai.
- Browser menyimpan sesi lama.

Jawaban AI:
Logout lalu login ulang. Jika menu tetap tidak muncul, hubungi IT karena kemungkinan hak akses atau role user belum sesuai.

Masalah: Access denied.
Kemungkinan penyebab:
- User tidak punya permission untuk menu tersebut.
- Role tidak cocok dengan menu.
- Company access tidak sesuai.

Jawaban AI:
Ini biasanya bukan error input user. Minta user hubungi IT atau administrator JIFAS untuk pengecekan role dan menu access.

Masalah: Session expired atau sering logout.
Kemungkinan penyebab:
- Sesi login habis.
- Token expired.
- Browser idle terlalu lama.
- Network atau API bermasalah.

Jawaban AI:
Coba login ulang. Jika sering terjadi dalam waktu pendek, hubungi IT.

Masalah: Forgot password tidak mengirim email.
Kemungkinan penyebab:
- Email salah.
- Email belum terdaftar.
- Mail server delay.
- Akun dikelola oleh Active Directory.

Jawaban AI:
Cek spam atau junk mail. Jika tidak ada, hubungi IT untuk reset akun.


4. Modul Accounting

#4.1 Fungsi Modul Accounting

Modul Accounting adalah pusat pencatatan transaksi keuangan di JIFAS. Semua transaksi yang berasal dari invoice, payment, cash bank, receiving, PUM, sales receive, atau transaksi manual pada akhirnya harus tercermin di jurnal accounting.

Fungsi utama:
- Mencatat jurnal harian.
- Membuat jurnal memorial.
- Membuat jurnal kas CAJE atau PAJE.
- Menyesuaikan jurnal melalui adjust.
- Membatalkan jurnal melalui reverse.
- Melakukan posting ke ledger.
- Menghasilkan laporan keuangan.
- Membantu audit trail.

Modul Accounting harus menjaga prinsip double entry. Setiap jurnal harus balance, yaitu total debit sama dengan total credit.


#4.2 Prinsip Double Entry

Dalam accounting, setiap transaksi harus memiliki sisi debit dan credit. Total debit harus sama dengan total credit. Jika tidak sama, jurnal tidak boleh disimpan atau tidak boleh diproses lanjut.

Contoh benar:
Debit Beban Gaji 10.000.000
Credit Bank 10.000.000
Total debit 10.000.000 sama dengan total credit 10.000.000.

Contoh salah:
Debit Beban Gaji 10.000.000
Credit Bank 9.500.000
Selisih 500.000. Jurnal tidak balance dan harus diperbaiki.

AI harus menjelaskan bahwa pesan “Total Debit and Credit must be equal” bukan error sistem, tetapi aturan akuntansi yang wajib dipenuhi.


#4.3 Status Jurnal

Status jurnal menunjukkan posisi dokumen dalam siklus accounting.

Unchecked adalah jurnal baru atau belum diperiksa.
Checked adalah jurnal sudah diperiksa dan siap diposting.
Posted adalah jurnal sudah masuk ledger dan menjadi final.
Void adalah jurnal dibatalkan.
Need Reverse adalah jurnal perlu dibuatkan jurnal pembalik.
Need Complete Void adalah jurnal perlu pembatalan penuh.
Removed adalah jurnal sudah dihapus atau tidak digunakan lagi.

Aturan penting:
- Jurnal Posted tidak boleh diubah langsung.
- Jika jurnal Posted salah, gunakan Adjust atau Reverse.
- Jangan delete jurnal posted karena audit trail harus tetap ada.


#4.4 Master Data Accounting

Master data adalah data referensi yang dipakai semua proses accounting.

Company adalah perusahaan yang sedang diproses datanya.
COA atau Chart of Account adalah daftar akun accounting.
Division adalah divisi organisasi.
Department adalah departemen di dalam divisi.
Vendor adalah pihak supplier atau creditor.
Tenant atau Customer adalah pihak debtor atau pelanggan.
Object Code adalah klasifikasi tambahan seperti project, cost center, region, unit, atau objek biaya.
GL Type adalah jenis jurnal atau tipe transaksi general ledger.
Active Period adalah periode accounting yang sedang dibuka.

Jika master data tidak muncul, bukan user yang harus memperbaiki. Arahkan ke Accounting Admin atau IT.


#4.5 Active Period

Active Period adalah rentang tanggal yang diperbolehkan untuk input transaksi accounting. Sistem membatasi tanggal jurnal agar hanya berada dalam periode aktif.

Jika muncul pesan bahwa company tidak memiliki active period, artinya periode accounting belum dibuka atau tidak tersedia. User harus menghubungi Accounting Admin atau IT Administrator.

AI harus menjawab:
Sistem tidak bisa menerima jurnal di luar periode aktif. Silakan konfirmasi periode aktif ke Accounting atau hubungi IT jika periode seharusnya sudah dibuka tetapi tetap tidak muncul.


5. Journal Transaction

#5.1 Fungsi Journal Transaction

Journal Transaction adalah halaman utama untuk melihat, mencari, memeriksa, membuat, menyesuaikan, membatalkan, dan mengekspor jurnal. Halaman ini menjadi pusat kerja accounting staff dan accounting supervisor.

User dapat:
- Mencari jurnal berdasarkan nomor dokumen.
- Mencari jurnal berdasarkan rentang tanggal.
- Memfilter berdasarkan status.
- Melihat detail jurnal.
- Membuat Journal Memorial.
- Membuat CAJE atau PAJE.
- Melakukan Adjust.
- Melakukan Reverse.
- Export data ke Excel.


#5.2 Cara Mencari Jurnal

Ada dua cara umum mencari jurnal.

Cara pertama adalah mencari berdasarkan Document Number. User memilih filter document number lalu memasukkan nomor jurnal atau nomor dokumen. Cara ini cocok jika user sudah tahu nomor jurnal.

Cara kedua adalah mencari berdasarkan Date Range. User memasukkan tanggal awal dan tanggal akhir, lalu klik Preview. Cara ini cocok jika user hanya tahu periode transaksi.

Jika jurnal tidak muncul:
- Pastikan tanggal yang dipilih benar.
- Pastikan nomor dokumen tidak salah ketik.
- Coba hapus filter status.
- Klik Preview setelah mengubah parameter.
- Pastikan company yang dipilih benar.
- Jika tetap tidak muncul, tanyakan ke accounting supervisor atau IT.


#5.3 Data yang Ditampilkan di Journal Transaction

Data utama yang biasa tampil:
- Doc No atau nomor jurnal.
- Created By atau pembuat jurnal.
- Last Update atau tanggal update terakhir.
- Doc Date atau tanggal jurnal.
- Amount atau nilai jurnal.
- Bank jika terkait transaksi bank.
- Description atau deskripsi.
- Ref Doc atau dokumen referensi.
- Ref Num atau nomor referensi.
- Journal Description.
- Status.
- Action.

Detail jurnal berisi:
- Division.
- Department.
- COA.
- COA Description.
- Description.
- Object Code.
- Object Description.
- Debit.
- Credit.
- From.
- To.
- Status.


#5.4 Alur Kerja Journal Transaction

1. Accounting staff membuat jurnal.
2. Sistem memberi status awal seperti Unchecked.
3. Supervisor atau pihak berwenang memeriksa jurnal.
4. Jika benar, jurnal menjadi Checked.
5. Jurnal Checked dapat diposting.
6. Setelah Posted, jurnal menjadi final dan masuk ledger.
7. Jika ada kesalahan setelah posted, koreksi dilakukan dengan Adjust atau Reverse.


#5.5 Error Handling Journal Transaction

Masalah: Jurnal tidak ketemu.
Solusi: Periksa company, filter tanggal, nomor dokumen, dan status. Jika tetap tidak ditemukan, hubungi accounting supervisor atau IT.

Masalah: Jurnal tidak bisa diedit.
Solusi: Jika jurnal sudah Posted, ini normal. Gunakan Adjust atau Reverse sesuai kebutuhan dan ikuti approval internal.

Masalah: Data gagal dimuat.
Solusi: Login ulang dan cek koneksi. Jika masih gagal, hubungi IT karena kemungkinan API atau token bermasalah.

Masalah: Export tidak bisa.
Solusi: Pastikan data sudah tampil sebelum export. Jika file tetap gagal diunduh, coba browser lain atau hubungi IT.


6. Journal Memorial

#6.1 Fungsi Journal Memorial

Journal Memorial adalah jurnal manual untuk transaksi yang tidak melalui alur operasional normal. Biasanya dipakai untuk adjustment akhir periode, koreksi pembukuan, accrual, depreciation, reserve, dan penyesuaian lain.

Contoh penggunaan:
- Penyesuaian depreciation akhir bulan.
- Koreksi akun yang salah.
- Jurnal accrual biaya.
- Penyesuaian inventory.
- Pencadangan piutang tak tertagih.


#6.2 Alur Membuat Journal Memorial

1. User membuka Journal Transaction.
2. User memilih Create Journal Memorial.
3. Sistem menampilkan form jurnal memorial.
4. User memilih tanggal sesuai active period.
5. User mengisi deskripsi.
6. User dapat mengisi Ref Num dan Ref Doc jika ada.
7. User memilih vendor atau tenant jika akun membutuhkan party.
8. User menambahkan detail baris jurnal.
9. User memilih divisi, departemen, COA, object code jika diperlukan, debit, dan credit.
10. Sistem menghitung total debit dan credit.
11. User memastikan total debit sama dengan total credit.
12. User menekan Save.
13. Sistem menyimpan jurnal jika semua validasi terpenuhi.


#6.3 Field Penting Journal Memorial

Date adalah tanggal jurnal dan wajib diisi.
Amount dihitung otomatis dari total debit.
Ref Num adalah nomor referensi, opsional.
Ref Doc adalah dokumen sumber, opsional.
Description wajib diisi dan harus jelas.
Vendor diisi jika akun membutuhkan vendor.
Tenant atau Customer diisi jika akun membutuhkan tenant atau customer.
Division wajib pada detail.
Department wajib pada detail.
COA wajib pada detail.
Object Code opsional, tergantung kebutuhan klasifikasi.
Debit atau Credit diisi sesuai posisi transaksi.


#6.4 Validasi Journal Memorial

Sistem mengecek:
- Date harus diisi.
- Description harus diisi.
- Minimal ada satu baris detail.
- Total debit harus sama dengan total credit.
- Amount harus lebih besar dari nol.
- Vendor harus diisi jika COA membutuhkan vendor.
- Tenant atau customer harus diisi jika COA membutuhkan tenant atau customer.
- COA harus tersedia.
- Object Code harus dipilih jika transaksi membutuhkan object.

Jika validasi gagal, sistem menolak save dan menampilkan pesan.


#6.5 Error Handling Journal Memorial

Pesan: Total Debit and Credit must be equal.
Arti: Jurnal belum balance.
Solusi: Periksa semua nominal debit dan credit sampai totalnya sama.

Pesan: Description must be filled.
Arti: Deskripsi kosong.
Solusi: Isi deskripsi yang jelas.

Pesan: Date must be filled.
Arti: Tanggal belum dipilih.
Solusi: Pilih tanggal sesuai active period.

Pesan: Total amount must be greater than zero.
Arti: Tidak ada nominal yang valid.
Solusi: Isi nominal debit atau credit.

Pesan: Vendor must be filled atau Failed Please fill in the Vendor Code.
Arti: COA membutuhkan vendor tetapi vendor belum dipilih.
Solusi: Pilih vendor. Jika vendor tidak ada, hubungi Accounting Admin atau IT.

Pesan: Tenant Customer must be filled atau Failed Please fill in the Tenant Customer Code.
Arti: COA membutuhkan tenant atau customer tetapi belum dipilih.
Solusi: Pilih tenant atau customer. Jika tidak tersedia, hubungi Accounting Admin atau IT.

Pesan: Company does not have active period.
Arti: Tidak ada active period untuk company tersebut.
Solusi: Hubungi Accounting Admin atau IT Administrator.


7. CAJE dan PAJE

#7.1 Fungsi CAJE dan PAJE

CAJE dan PAJE adalah jurnal kas. Modul ini dipakai untuk membuat jurnal yang berkaitan dengan kas masuk atau kas keluar.

CAJE umumnya berkaitan dengan cash adjustment atau cash journal.
PAJE umumnya berkaitan dengan payment adjustment atau payment journal.

Nama detailnya dapat mengikuti konfigurasi GL Type perusahaan. User memilih tipe jurnal dari daftar GL Type yang tersedia.


#7.2 Perbedaan Journal Memorial dan CAJE PAJE

Journal Memorial lebih umum dan dapat dipakai untuk berbagai jurnal penyesuaian.
CAJE PAJE lebih spesifik untuk transaksi kas atau jurnal dengan GL Type tertentu.

Keduanya tetap memakai prinsip double entry. Total debit harus sama dengan total credit.


#7.3 Alur Membuat CAJE PAJE

1. User membuka Journal Transaction.
2. User memilih Create Caje Paje.
3. User memilih tipe jurnal atau GL Type.
4. User memilih tanggal.
5. User mengisi deskripsi.
6. User mengisi Ref Num dan Ref Doc jika ada.
7. User menambahkan detail baris jurnal.
8. User memilih division, department, COA, object code jika diperlukan, debit, dan credit.
9. Sistem menghitung total.

8. Adjust Journal

#8.1 Fungsi Adjust Journal

Adjust Journal digunakan untuk melakukan koreksi terhadap jurnal yang sudah ada tanpa menghapus jurnal lama. Tujuannya menjaga audit trail. Adjustment membuat catatan koreksi, sehingga auditor dapat melihat jurnal awal dan jurnal penyesuaiannya.

Adjust digunakan jika:
- Ada salah akun.
- Ada selisih nominal.
- Ada koreksi dari supervisor.
- Ada kebutuhan menambahkan transaksi koreksi tanpa membatalkan penuh jurnal lama.


#8.2 Alur Adjust Journal

8. User menyimpan adjustment.

9. Reverse Journal

#9.1 Fungsi Reverse Journal

Reverse Journal digunakan untuk membatalkan efek jurnal dengan membuat jurnal kebalikan. Debit pada jurnal lama menjadi credit pada jurnal reverse, dan credit pada jurnal lama menjadi debit pada jurnal reverse.

Reverse bukan delete. Reverse adalah cara accounting yang benar untuk membatalkan jurnal yang sudah posted atau tidak boleh diubah langsung.


#9.2 Kapan Reverse Digunakan

Reverse digunakan jika:
- Jurnal posted ternyata salah total.
- Transaksi harus dibatalkan penuh.
- Jurnal dibuat ganda.
- Dokumen sumber ternyata tidak valid.
- Supervisor meminta pembatalan dengan audit trail.


#9.3 Alur Reverse Journal

1. User mencari jurnal yang akan dibalik.
2. User memilih action Reverse.
3. Sistem menampilkan data jurnal asli.
4. User memilih Reverse Date.
5. Sistem mengisi atau user mengisi Description.
6. Sistem menampilkan detail jurnal lama.
7. User mengonfirmasi reverse.
8. Sistem membuat jurnal reverse baru.
9. Jurnal asli tetap ada sebagai audit trail.
10. Net effect accounting menjadi nol.


#9.4 Error Handling Reverse Journal

Pesan: Cannot continue to save. Please fill mandatory field before.
Solusi: Isi Reverse Date dan Description.

Pesan: Cannot get data journal. Please contact your IT Administrator.
Arti: Sistem gagal mengambil data jurnal referensi.
Solusi: Hubungi IT Administrator dengan nomor jurnal dan screenshot error.

Pesan: Request failed atau server error.
Solusi: Hubungi IT karena reverse membutuhkan koneksi API yang valid.

Jika reverse berhasil, sistem menampilkan pesan bahwa Journal Reverse berhasil dibuat.

#20.1 Total Debit and Credit must be equal

Penyelesaian:
- Hitung ulang total debit.
- Hitung ulang total credit.
- Pastikan tidak ada salah input angka.
- Pastikan angka tidak masuk ke debit dan credit sekaligus pada baris yang sama.
- Pastikan tidak ada baris kosong.
- Simpan setelah balance.

Eskalasi ke IT hanya jika angka di layar sudah sama tetapi sistem tetap menolak.

- Login ulang.
- Jika tetap muncul, hubungi IT untuk pengecekan role dan token.


#20.12 Request Failed atau Internal Server Error

Arti:
- API error.
- Server error.
- Koneksi bermasalah.
- Data input mungkin menyebabkan error backend.

Penyelesaian:
- Screenshot error.
- Catat menu, company, periode, dan dokumen.
- Hubungi IT.

22. Contoh Tanya Jawab untuk AI Chatbot

Pertanyaan: Saya tidak bisa login JIFAS.
Jawaban: Coba pastikan username dan password sudah benar serta Caps Lock tidak aktif. Jika akun Windows juga bermasalah atau pesan error tetap muncul, hubungi IT JIFAS karena kemungkinan akun, password, atau akses user perlu dicek.

Pertanyaan: Saya bisa login tapi menu Accounting tidak muncul.
Jawaban: Kemungkinan role atau menu access belum diberikan. Logout dan login ulang terlebih dahulu. Jika tetap tidak muncul, hubungi IT untuk pengecekan role dan akses menu.

Pertanyaan: Kenapa muncul Total Debit and Credit must be equal?
Jawaban: Artinya jurnal belum balance. Dalam accounting, total debit harus sama dengan total credit. Periksa semua baris nominal dan pastikan total debit sama dengan total credit sebelum save.

Pertanyaan: Kenapa vendor harus diisi?
Jawaban: COA yang dipilih membutuhkan vendor sebagai party. Pilih vendor dari dropdown. Jika vendor tidak tersedia, hubungi Accounting Admin atau IT untuk pengecekan master vendor.

Pertanyaan: Kenapa tenant customer harus diisi?
Jawaban: COA yang dipilih membutuhkan tenant atau customer. Pilih tenant atau customer dari dropdown. Jika tidak ada, hubungi Accounting Admin atau IT.

Pertanyaan: Kenapa jurnal posted tidak bisa diedit?
Jawaban: Jurnal posted sudah final dan masuk ledger. Untuk koreksi, gunakan Adjust jika hanya perlu koreksi, atau Reverse jika transaksi harus dibatalkan penuh. Jangan menghapus jurnal posted.

Pertanyaan: Apa bedanya Adjust dan Reverse?
Jawaban: Adjust digunakan untuk membuat koreksi tambahan tanpa membatalkan penuh jurnal lama. Reverse digunakan untuk membuat jurnal kebalikan sehingga efek jurnal lama menjadi nol.

Pertanyaan: Kenapa report kosong?
Jawaban: Pastikan company dan periode sudah dipilih, lalu klik Preview. Jika data seharusnya ada tetapi tetap kosong, hubungi Accounting Supervisor atau IT.

Pertanyaan: Kenapa tidak bisa export Excel?
Jawaban: Pastikan data sudah tampil terlebih dahulu. Jika tabel masih kosong, export tidak bisa dilakukan. Jika data sudah tampil tapi export gagal, hubungi IT.

Pertanyaan: Apa itu Aging AP?
Jawaban: Aging AP adalah laporan umur hutang vendor. Laporan ini membantu finance melihat hutang berdasarkan umur jatuh tempo, misalnya current, overdue 1 sampai 30 hari, dan overdue lebih dari 365 hari.

Pertanyaan: Apa itu Aging AR?
Jawaban: Aging AR adalah laporan umur piutang customer. Laporan ini membantu finance melihat piutang berdasarkan umur jatuh tempo dan menentukan prioritas penagihan.

Pertanyaan: Kenapa active period tidak tersedia?
Jawaban: Company tersebut belum memiliki periode accounting aktif atau sistem gagal mengambil data periode. Hubungi Accounting Admin atau IT Administrator.

Pertanyaan: Bulk posting gagal.
Jawaban: Pastikan dokumen yang dipilih memang siap posting. Jika pesan gagal berasal dari validasi dokumen, perbaiki dokumennya. Jika pesannya berupa server error atau request failed, hubungi IT.

Pertanyaan: Send to consolidation gagal.
Jawaban: Pastikan company dan period benar, dan Trial Balance sudah dipreview. Jika tetap gagal, hubungi IT karena proses ini melibatkan integrasi consolidation.

Pertanyaan: Cancel Purchase tidak bisa disimpan.
Jawaban: Pastikan cancellation date, cancellation type, reason, dan funding allocation sudah diisi. Jika field handover to atau change to muncul, field tersebut juga wajib diisi.

Pertanyaan: Mark as Sales Revenue tidak bisa disimpan.
Jawaban: Pastikan tanggal dan funding allocation sudah lengkap. Jika COA, division, atau department tidak muncul, hubungi Accounting Admin atau IT.


23. Panduan Eskalasi ke IT

AI harus menyarankan user mengirim informasi berikut ke IT:
- Nama user dan username.
- Company yang dipilih.
- Menu yang diakses.
- Nomor dokumen atau jurnal jika ada.
- Periode atau tanggal yang dipilih.
- Screenshot error.
- Waktu kejadian.
- Browser yang digunakan.
- Apakah sudah logout dan login ulang.
- Apakah user lain mengalami masalah yang sama.

Contoh pesan ke IT:
Saya mengalami error di JIFAS pada menu Journal Transaction. Company KI, nomor jurnal JM-2026-001, periode 01/02/2026 sampai 28/02/2026. Error yang muncul: Cannot get data journal. Saya sudah logout dan login ulang tetapi masih sama. Terlampir screenshot error.


24. Batasan Jawaban AI

AI boleh membantu menjelaskan alur, validasi, istilah, dan langkah troubleshooting.

AI tidak boleh:
- Menyuruh user menghapus jurnal posted.
- Menyuruh user bypass approval.
- Menyuruh user mengubah database.
- Menebak role user tanpa data.
- Menjamin transaksi benar tanpa verifikasi accounting.
- Mengganti keputusan supervisor, finance, tax, accounting, atau auditor.

Jika pertanyaan membutuhkan otorisasi, AI harus mengarahkan user ke pihak berwenang.

Ledger bertanggung jawab atas buku besar per COA.
