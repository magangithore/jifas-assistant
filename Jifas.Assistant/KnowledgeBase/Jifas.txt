JIFAS — Jababeka Integrated Finance Accounting System
=======================================================
Dokumen: Overview & Referensi Utama JIFAS
Berlaku untuk: Seluruh unit bisnis Jababeka Group

=======================================================
1. APA ITU JIFAS?
=======================================================

JIFAS adalah sistem ERP keuangan terintegrasi berbasis web milik PT Jababeka Tbk.
Digunakan oleh seluruh unit bisnis Jababeka Group untuk mengelola proses keuangan,
akuntansi, anggaran, pembayaran, dan pelaporan secara terpusat dan real-time.

JIFAS menggantikan JIFAS Desktop (digunakan sejak 2005, pengembangan dihentikan).
Semua user JIFAS Desktop diharuskan migrasi ke JIFAS Web.

4. MODUL-MODUL UTAMA JIFAS
=======================================================

--- 4.1 MASTER DATA ---
Pusat referensi data yang dipakai seluruh modul operasional JIFAS.

Sub-modul:
- Company       : Data perusahaan dan cabang
- Employee      : Data karyawan (dipakai di PUM, payroll)
- Vendor        : Data supplier / rekanan bisnis
- Division      : Struktur divisi organisasi
- Department    : Struktur departemen
- COA           : Chart of Accounts (kode akun keuangan)
- Account Period: Buka/tutup periode akuntansi bulanan/tahunan
- List COA      : Daftar COA aktif per periode
- Report Setup  : Mapping COA ke template laporan keuangan
- Budget        : Input dan kelola anggaran per COA/departemen
- Roles         : Kelola hak akses user

--- 4.2 INVOICE ---
Modul pengajuan tagihan/biaya dari creator ke approval chain.

Alur: Create -> Finance Checking -> Head Approval -> Tax -> Posting

Status dokumen:
- Draft            : Belum disubmit, masih bisa diedit
- Submitted        : Sudah disubmit ke Finance
- Finance Checking : Sedang diverifikasi Finance
- Head Approval    : Sedang menunggu approval Head/Director
- Approved         : Sudah disetujui semua pihak
- Posted           : Sudah diposting ke GL (TIDAK bisa diedit)
- Rejected         : Dikembalikan ke pembuat untuk diperbaiki
- Void             : Dibatalkan permanen

Tombol utama: Save | Submit | Approve | Reject | Tax Correction | Post | Void

--- 4.3 PUM (Perjalanan Uang Muka / Perjalanan Dinas) ---
Modul pengajuan uang muka operasional/perjalanan dinas karyawan.

Alur: Pengajuan -> Supervisor Approval -> Finance Approval -> Distribusi -> Realisasi -> Settlement

Status dokumen:
- Draft       : Belum disubmit
- Submitted   : Diajukan ke Supervisor
- Approved    : Disetujui semua pihak
- Distributed : Uang muka sudah dicairkan
- Realization : Karyawan sedang melaporkan pengeluaran aktual
- Settled     : Proses selesai, realisasi dikonfirmasi
- Rejected    : Ditolak, dikembalikan ke pembuat

Settlement:
- Realisasi < Uang Muka -> karyawan kembalikan sisa ke perusahaan
- Realisasi > Uang Muka -> perusahaan bayar kekurangannya

Role khusus:
- USRL : Bisa memilih department/divisi orang lain saat buat PUM
- USER : Hanya bisa ajukan PUM untuk diri sendiri

Sub-tab PUM:
- PPUM (Pengajuan PUM) : Buat dan kelola pengajuan uang muka
- Realization          : Input pengeluaran aktual setelah dinas

--- 4.4 RECEIVING ---
Modul penerimaan barang/jasa dari supplier atau internal.

Sub-modul:
- Receive of Sales           : Penerimaan dari transaksi penjualan
- Receive Tax (Tax Approval) : Penerimaan dengan aspek perpajakan
- Approval of Unidentified RV: Approval penerimaan tidak teridentifikasi

Catatan penting:
- Receive Tax WAJIB: NPWP lengkap + alamat WP harus diisi sebelum approve
- Jika tax rate salah -> Reject, lalu buat dokumen baru
- RV = Receive Voucher (nomor dokumen penerimaan)

Status: Need Finance Checking -> Approved -> Posted

--- 4.5 PAYMENT ---
Modul pembayaran atas invoice dan PUM yang sudah disetujui.

Sub-modul:
- Payment Invoice : Bayar tagihan vendor/supplier
- Payment PUM     : Cairkan/distribusikan uang muka PUM
- Payment Tax     : Pembayaran terkait kewajiban pajak
- List BG         : Daftar Bank Garansi yang tersedia

Alur: Finance Approval -> Head Approval -> Posting

Metode pembayaran: Transfer | BG (Bank Garansi) | Cek | Giro

--- 4.6 ACCOUNTING ---
Modul pencatatan jurnal dan laporan akuntansi.

Sub-modul:
- GL (General Ledger)    : Buku besar, jurnal manual (Journal Memorial)
- AP (Account Payable)   : Manajemen hutang ke vendor/supplier
- AR (Account Receivable): Manajemen piutang dari pelanggan
- Posting                : Rekam transaksi ke buku besar
- Bulk Posting           : Posting banyak dokumen sekaligus
- Inquiry AP/AR/CB/PUM   : Lihat saldo dan detail transaksi
- Konsolidasi            : Gabung laporan multi-cabang/divisi
- Acc Period             : Buka/tutup periode akuntansi

Tipe jurnal:
- Journal Transaction : Jurnal harian standar
- Journal Memorial    : Jurnal penyesuaian/koreksi manual
- CAJE                : Jurnal kas masuk
- PAJE                : Jurnal kas keluar

--- 4.7 REPORT ---
Modul laporan dan analisis keuangan.

Laporan Budget:
- Budget Card          : Kartu anggaran per cost center / COA
- Budget Committed     : Komitmen anggaran yang belum terealisasi
- Budget Payment       : Realisasi pembayaran vs anggaran
- Budget Realization   : Laporan realisasi anggaran
- Budget Receive       : Penerimaan terkait anggaran
- Committed Realization: Komitmen vs realisasi aktual

Laporan Kas/Bank:
- Daily Cashflow       : Arus kas harian
- Cashbank Detail      : Detail transaksi kas dan bank
- Cashbank Recap       : Rekap kas dan bank per periode
- Saldo Buku Bank      : Rekonsiliasi buku perusahaan vs bank
- Deposito Aktif       : Daftar deposito yang sedang berjalan

Laporan AP/AR/PUM:
- Inquiry AP           : Saldo hutang ke vendor
- Inquiry AR           : Saldo piutang dari customer
- Inquiry CB           : Saldo kas dan bank
- Inquiry PUM          : Saldo uang muka karyawan
- Realisasi PUM        : Laporan realisasi pengeluaran dinas

--- 4.8 OVER BUDGET ---
Muncul ketika nominal transaksi melebihi batas anggaran yang tersedia.

Status budget:
- OK : Nominal dalam batas anggaran -> lanjut normal
- CM : Cross Month — melebihi bulan ini, masih tersedia bulan depan
       Tindakan: Minta approval Supervisor/Finance
- CA : Cross Accumulation — melebihi akumulasi beberapa bulan ke depan
       Tindakan: Minta approval Head Finance / Management
- CY : Cross Year — melebihi total budget tahun ini
       Tindakan: SISTEM BLOKIR -> butuh approval CFO/Director

Penanganan over budget:
1. Kurangi nominal pengajuan agar sesuai budget
2. Gunakan COA alternatif yang budgetnya mencukupi
3. Minta approval exception dari level manajemen yang sesuai
4. Atau ajukan amendment budget ke Finance

=======================================================
5. ALUR APPROVAL UMUM JIFAS
=======================================================

Creator (buat & submit)
    -> Finance (verifikasi data, COA, budget, pajak)
    -> Head / Director (persetujuan final)
    -> Tax (jika ada aspek perpajakan)
    -> Posting ke GL (rekam ke buku besar — tidak bisa diedit lagi)

=======================================================
6. ROLES DAN HAK AKSES
=======================================================

WMTR (Webmaster)
  : IT admin — akses penuh ke semua modul dan konfigurasi sistem

USER (User)
  : Pengguna umum — akses sesuai modul yang ditetapkan

USRL (User All Level)
  : Seperti USER, tapi bisa memilih dept/divisi orang lain di PUM

FINA (Finance)
  : Staff finance — view semua transaksi, proses approval finance

SPVR (Supervisor Finance)
  : Approval tingkat supervisor keuangan

HDFA (Head Finance)
  : Approval tingkat kepala keuangan

TAXS (Tax)
  : Staf pajak — akses modul Tax Approval

FRPT (Finance Report)
  : Akses laporan keuangan saja (read-only)

Catatan penting:
- WMTR hanya diberikan ke tim IT, JANGAN berikan ke non-IT
- USRL punya hak istimewa memilih departemen orang lain saat buat PUM
- Jika user tidak punya akses modul tertentu -> minta ke admin/IT

=======================================================
7. ATURAN PENTING SISTEM
=======================================================

1. Dokumen sudah POSTED -> tidak bisa diedit atau dihapus
2. Periode akuntansi sudah DITUTUP -> tidak bisa input transaksi baru
3. REJECT -> dokumen kembali ke creator untuk diperbaiki lalu re-submit
4. VOID -> pembatalan permanen, dokumen tidak bisa diaktifkan kembali
5. Budget CY -> sistem MEMBLOKIR penyimpanan, butuh CFO/Director approval
6. NPWP dan alamat WP wajib lengkap sebelum Receive Tax bisa di-approve
7. COA yang sudah dipakai dalam transaksi TIDAK bisa dihapus
8. Laporan tidak muncul? Pastikan COA sudah di-mapping di Report Setup
9. Data tidak muncul di laporan? Pastikan jurnal sudah di-POSTING (bukan Draft)
